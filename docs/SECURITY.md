# Security Guide

## Security Layers

This stack implements defense-in-depth across all layers:

```
Layer 1: Network    → Traefik TLS, internal-only MySQL/Redis
Layer 2: Container  → Resource limits, non-root, health checks
Layer 3: PHP        → disable_functions, expose_php=Off, secure cookies
Layer 4: WordPress  → mu-plugin hardening (9 features), non-default prefix
Layer 5: Scripts    → set -euo pipefail, quoted variables, MYSQL_PWD
Layer 6: CI         → PHP lint, PHPCS, ESLint, PHPStan on every push
Layer 7: Git        → Pre-commit hooks, safety check hooks
```

## Network Security

### MySQL
- Exposed **only** on `127.0.0.1:{PORT}` — never on `0.0.0.0`
- Separate root and application user credentials
- Health check uses `MYSQL_PWD` env var (never CLI password)

### Redis
- Password-protected (`requirepass` in compose command)
- Not exposed externally — internal network only
- 64MB memory limit with LRU eviction

### Traefik
- TLS termination at the proxy level
- BasicAuth middleware for phpMyAdmin
- Host-based routing (no path collisions)

## PHP Hardening

The custom Dockerfile includes `security.ini`:

```ini
expose_php = Off
display_errors = Off
log_errors = On
session.cookie_httponly = On
session.cookie_secure = On
session.use_strict_mode = On
disable_functions = exec,passthru,shell_exec,system,proc_open,popen,parse_ini_file,show_source
```

## WordPress mu-plugin

The `{slug}-security.php` mu-plugin automatically:

1. **Disables XML-RPC** — blocks xmlrpc.php entirely
2. **Restricts REST API** — unauthenticated users get 401
3. **Removes versions** — strips WordPress/plugin version strings
4. **Disables file editor** — no Theme/Plugin Editor in admin
5. **Cleans headers** — removes Windows Live Writer, shortlink, feed links
6. **Blocks author enumeration** — redirects `?author=N` queries
7. **Adds security headers** — X-Frame-Options, X-Content-Type-Options, X-XSS-Protection, Referrer-Policy, Permissions-Policy
8. **Restricts app passwords** — only administrators can use them
9. **Rate limits login** — 5 attempts per 15m window per IP (X-Forwarded-For aware)

## Coding Security Standards

### PHP (WordPress)

```php
// INPUT: Always sanitize
$name = sanitize_text_field($_POST['name']);
$id = absint($_GET['id']);
$html = wp_kses_post($_POST['content']);

// OUTPUT: Always escape
echo esc_html($name);
echo esc_attr($value);
echo esc_url($link);

// SQL: Always prepare
$wpdb->get_results($wpdb->prepare(
    "SELECT * FROM {$wpdb->prefix}table WHERE id = %d AND status = %s",
    $id,
    $status
));

// CSRF: Always verify nonces
wp_verify_nonce($_POST['_nonce'], 'action_name');
```

### Shell Scripts

```bash
#!/bin/bash
set -euo pipefail          # Exit on error, undefined vars, pipe failures

# Quote all variables
echo "File: ${BACKUP_FILE}"

# Use MYSQL_PWD for credentials (never -p on CLI)
export MYSQL_PWD="${MYSQL_ROOT_PASSWORD}"
mysqldump -u root "$DB_NAME"

# Validate .env before operations
if [[ ! -f .env ]]; then
    echo "ERROR: .env file not found"
    exit 1
fi
```

### JavaScript

```javascript
// Never use innerHTML with untrusted data
element.textContent = userInput;  // Safe
element.innerHTML = userInput;    // DANGEROUS

// Use DOMPurify if HTML is required
element.innerHTML = DOMPurify.sanitize(userInput);
```

## Secrets Management

| Secret | Location | How Generated |
|--------|----------|--------------|
| MySQL root password | `.env` | `openssl rand -base64 32` |
| MySQL user password | `.env` | `openssl rand -base64 32` |
| Redis password | `.env` | `openssl rand -base64 32` |
| WordPress salts (8) | `.env` | `openssl rand -base64 64` |
| PMA BasicAuth | `.env` | User-defined |

All secrets live in `.env` which is:
- Generated by `setup.sh` with cryptographically secure random values
- Listed in `.gitignore` (never committed)
- `.env.example` contains variable names only (no values)

## CI Security Checks

The CI pipeline includes:
- **PHP Lint**: Syntax validation for all PHP files
- **PHPCS/WPCS**: WordPress coding standards (includes security sniffs)
- **PHPStan**: Static analysis to catch type errors and unsafe patterns
- **ESLint**: JavaScript linting
- **Docker Compose Validation**: Ensures compose file is valid

## Pre-commit Hooks

Husky runs on every commit:
1. PHP lint check on staged `.php` files
2. ESLint on staged `.js` files
3. TDD reminder (to write tests first)

## Audit Checklist

Run periodically with `/security-audit`:

- [ ] `.env` not in git: `git ls-files .env`
- [ ] MySQL not on 0.0.0.0: `grep "0.0.0.0" docker-compose.yml`
- [ ] No hardcoded credentials: `grep -rn "password\|secret" --include="*.php" --include="*.sh"`
- [ ] WP_DEBUG is false in production
- [ ] Backups are not publicly accessible
- [ ] Redis has authentication enabled
- [ ] All scripts use `set -euo pipefail`
- [ ] File permissions are restrictive on sensitive files
